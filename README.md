# 台灣銀行交易資料清洗管線（Banking Data Cleaning Pipeline）
針對台灣銀行交易情境設計的資料清洗管線，涵蓋資料診斷（Profiling）→ 清理（Cleaning）→ 驗證（Validation）的完整流程，並整合日誌記錄（Logging）與命令列參數（CLI），可作為排程腳本自動執行。

## 專案目標
實務上的金融交易資料往往包含格式不一致、缺失值、重複資料及編碼問題。本專案模擬真實銀行交易資料的清理流程，設計可重複使用的資料清洗管線，並透過驗證機制與日誌監控，提升資料品質與流程可靠性。

## 模擬的資料品質問題
本專案使用合成資料，模擬台灣金融實務中常見的資料品質問題，以展示完整的資料清理（Data Cleaning）與前處理（Preprocessing）流程：
- **民國年與西元年混用：**  同一欄位同時包含 113/03/11 與 2024/03/11，需先辨識年份格式，再統一轉換為西元。
- **多種日期格式：** 包含西元斜線（YYYY/MM/DD）、民國斜線（YYY/MM/DD）、ISO（YYYY-MM-DD）、歐式（DD-MM-YYYY）、中文格式（YYYY年MM月DD日）。
- **前導零遺失風險：** 客戶ID（7碼）與分行代碼（3碼）以 dtype=str 讀取，避免被視為數值而遺失前導零。
- **全形字元混入：** 金額欄位可能包含全形數字（如 ８００），透過 Unicode Normalization（NFKC）統一格式。
- **多種空值表示方式：** 同時處理 NaN、N/A、None、-、空字串及空白字元等不同形式。
- **複合式金額格式：** 同一筆資料可能同時包含千分位逗號、NT$前綴及前後空白，需逐步清理後才能轉換為數值。
- **重複資料：** 區分完全重複列（技術事故）與業務重複資料（需人工判斷），採不同策略處理。

## 設計思路
- **先診斷，再清理：** 透過格式骨架分析（將數字替換為佔位符）盤點所有格式變體，再制定對應清理策略，避免遺漏未預期的資料型態。
- **依資料特性選擇處理方式：** 日期格式彼此互斥，因此採用遮罩（Mask）分流解析；金額欄位的資料品質問題可同時存在，因此以單一清理管線依序處理。
- **保留原始資料：** 所有清理結果寫入 _clean 新欄位，原始欄位保持唯讀，方便後續比對、驗證與除錯。

## 驗證設計
- **通則驗證（Assertions）：** 檢查所有批次皆應符合的規則，例如金額必須為正值、交易類型符合白名單、識別碼格式正確、交易日期不可晚於今天，以及不得存在完全重複資料；任一條件不符即中止流程。
- **批次監控（Logging）：** 記錄各批次的重要統計資訊，例如資料筆數、空值數量及業務重複筆數，作為跨批次監控與趨勢追蹤，不影響流程執行。
- **金額驗證：** 採用獨立雙路徑（Dual Validation）機制，清理管線與驗證流程以不同邏輯分別計算金額總和，僅在兩者結果一致時才視為驗證通過。

## 執行方式
```bash
pip install pandas

#使用預設路徑
python clean_transactions.py

#指定輸入輸出
python clean_transactions.py --input transactions_0711.csv --output clean_0711.csv
```
